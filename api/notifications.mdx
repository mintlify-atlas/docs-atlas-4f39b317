title: "Notifications API — Driver Inbox Endpoints"
sidebarTitle: "Notifications"
description: "REST endpoints to retrieve and mark in-app notifications for delivery drivers. Pass the user's numeric ID in the x-user-id header."
---

The Notifications API provides the in-app inbox for delivery drivers (`Repartidor` role). Notifications are created automatically by the Orders API — when a new order is assigned to a driver, or when an order's status changes from outside the driver's own actions. All endpoints identify the requesting user via the `x-user-id` request header rather than a Bearer token; omitting this header returns an empty result set rather than an error.

---

### GET /api/notificaciones

Returns up to 50 notifications for the specified user, ordered from most recent to oldest.

<Note>
  Pass the user's numeric ID in the `x-user-id` header. If the header is absent, the response will be an empty array.
</Note>

**Headers**

| Header | Type | Required | Description |
|--------|------|----------|-------------|
| `x-user-id` | number | Yes | Numeric ID of the user whose notifications to retrieve. |

**Example request**

```bash
curl http://localhost:3000/api/notificaciones \
  -H "x-user-id: 3"
```

**Example response**

```json
[
  {
    "id": 12,
    "ordenId": 7,
    "tipo": "nueva_orden",
    "titulo": "Nueva orden asignada",
    "mensaje": "Se te ha asignado la orden #7 para Tienda López.",
    "leida": false,
    "fecha": "2025-01-15T14:05:00.000Z"
  },
  {
    "id": 10,
    "ordenId": 5,
    "tipo": "estado_actualizado",
    "titulo": "Estado actualizado",
    "mensaje": "La orden #5 fue marcada como Entregado.",
    "leida": true,
    "fecha": "2025-01-14T09:30:00.000Z"
  }
]
```

<ResponseField name="id" type="number">Unique numeric ID of the notification.</ResponseField>
<ResponseField name="ordenId" type="number">ID of the order that triggered this notification.</ResponseField>
<ResponseField name="tipo" type="string">Machine-readable event type, e.g. `nueva_orden` or `estado_actualizado`.</ResponseField>
<ResponseField name="titulo" type="string">Short display title shown in the notification badge.</ResponseField>
<ResponseField name="mensaje" type="string">Full human-readable notification body.</ResponseField>
<ResponseField name="leida" type="boolean">`false` if the notification has not yet been viewed; `true` once marked as read.</ResponseField>
<ResponseField name="fecha" type="string">ISO 8601 timestamp of when the notification was created.</ResponseField>

---

### PUT /api/notificaciones/:id/leida

Marks a single notification as read for the requesting user.

<Note>
  Pass the user's numeric ID in the `x-user-id` header to confirm ownership before marking the notification.
</Note>

<ParamField path="id" type="number" required>
  Numeric ID of the notification to mark as read.
</ParamField>

**Headers**

| Header | Type | Required | Description |
|--------|------|----------|-------------|
| `x-user-id` | number | Yes | Numeric ID of the user who owns the notification. |

**Example request**

```bash
curl -X PUT http://localhost:3000/api/notificaciones/12/leida \
  -H "x-user-id: 3"
```

**Example response**

```json
{ "ok": true }
```

**Errors**

| Status | Reason |
|--------|--------|
| `404` | No notification found with the given ID for this user. |

---

### PUT /api/notificaciones/leer-todas

Marks all unread notifications as read for the requesting user in a single operation.

<Note>
  Pass the user's numeric ID in the `x-user-id` header. If the header is missing, the request has no effect and still returns `{ "ok": true }`.
</Note>

**Headers**

| Header | Type | Required | Description |
|--------|------|----------|-------------|
| `x-user-id` | number | Yes | Numeric ID of the user whose notifications to bulk-mark as read. |

**Example request**

```bash
curl -X PUT http://localhost:3000/api/notificaciones/leer-todas \
  -H "x-user-id: 3"
```

**Example response**

```json
{ "ok": true }
```
